# Contributing to the Autonomous Constitutional Framework

Thank you for your interest in contributing to ACF. This project benefits from the expertise of researchers, engineers, governance practitioners, and anyone invested in the responsible development of autonomous AI systems.

## Code of Conduct

All participants in this project are expected to abide by our [Code of Conduct](CODE_OF_CONDUCT.md). By contributing, you agree to uphold a respectful, inclusive, and harassment-free environment.

## How to Contribute

### Reporting Issues

If you find a bug, inconsistency, or area for improvement:

1. **Search existing issues** to avoid duplicates.
2. **Open a new issue** with a clear, descriptive title.
3. **Provide context** — include the relevant section, protocol number, or document path.
4. **Describe the expected vs. actual behavior** or the improvement you're suggesting.

> **Security issues:** Do **not** open public issues for security vulnerabilities. See [SECURITY.md](SECURITY.md) for responsible disclosure instructions.

### Suggesting Enhancements

We welcome proposals for new protocols, amendments to existing protocols, and framework extensions. To suggest an enhancement:

1. Open an issue with the `enhancement` label.
2. Describe the problem or gap the enhancement addresses.
3. Explain your proposed solution and how it aligns with ACF's constitutional principles.
4. Reference any supporting research or precedent.

### Submitting Pull Requests

1. **Fork the repository** and create a feature branch from `main`.
2. **Make your changes** in a focused, well-scoped commit (or series of commits).
3. **Follow existing conventions** — match the style, formatting, and structure of surrounding documents.
4. **Write clear commit messages** that explain *what* changed and *why*.
5. **Open a pull request** against `main` with:
   - A descriptive title (under 70 characters)
   - A summary of changes and motivation
   - References to any related issues (e.g., `Closes #42`)
6. **Respond to review feedback** promptly and constructively.

### Pull Request Review Process

All pull requests are reviewed for:

- **Accuracy** — Technical and conceptual correctness
- **Consistency** — Alignment with ACF principles and existing documentation
- **Clarity** — Readability and accessibility for the intended audience
- **Scope** — Changes are focused and do not introduce unrelated modifications

Reviews are typically completed within 7 business days. Complex proposals involving protocol amendments or constitutional changes may require extended review and community discussion, as described in [GOVERNANCE.md](GOVERNANCE.md).

## Areas Where Contributions Are Welcome

We especially welcome contributions in the following areas:

### Protocol Documentation
- Clarifications and improvements to existing protocol specifications
- Additional examples and use cases for protocol application
- Cross-references between related protocols

### Implementation Guides
- Integration patterns for specific AI platforms and architectures
- Reference implementations of ACF verification mechanisms
- Tooling and automation for governance compliance checking

### Research and Citations
- Academic papers referencing or extending ACF concepts
- Empirical analysis of constitutional governance in AI systems
- Comparative studies with other governance frameworks
- Mathematical proofs related to conservation law enforcement

### Translations
- Translations of core documentation into other languages
- Localization of governance terminology for different regulatory contexts
- Region-specific implementation guidance (e.g., EU AI Act compliance mapping)

### Testing and Verification
- Test cases for protocol compliance verification
- Benchmarks for entropy-based verification methods
- Cross-substrate verification test suites

## Style Guide

### Documentation
- Use clear, precise, academic-grade language.
- Avoid jargon unless it is defined within the framework.
- Use Markdown for all documentation files.
- Follow the existing heading hierarchy and document structure.
- Define acronyms on first use (e.g., "Human Imperative Metric (HIM)").
- Include references and citations where appropriate.

### Naming Conventions
- Protocol names follow the existing naming scheme (see `/docs/protocols/`).
- Use lowercase with hyphens for file names: `protocol-specification.md`
- Place files in the appropriate `/docs/` subdirectory.
- Avoid abbreviations unless they are widely established within the framework.

### Commit Messages
- Use the imperative mood ("Add protocol documentation" not "Added protocol documentation").
- Keep the subject line under 72 characters.
- Include a body for non-trivial changes explaining the rationale.
